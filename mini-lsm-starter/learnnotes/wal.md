1. 什么时候应该在引擎中调用 fsync？如果每次都 fsync（比如每次 put 请求都 fsync）会怎样？

应该只在用户主动调用 `sync()` 时才执行 `fsync`。这是引擎对外承诺数据已持久化到磁盘的唯一时机。如果每次 `put`/`delete` 都调用 `fsync`，吞吐量会急剧下降。因为 `fsync` 会强制操作系统将 page cache 刷写到稳定存储介质，这是一个同步阻塞操作。这样做会让所有写操作退化为同步 I/O，完全丧失了 `BufWriter` 批量写入的优势。原本每秒数十万次的写操作可能会被压缩到只有几千次。一个常见的优化是**组提交（group commit）**：把多个写操作攒在一起，对整批数据只执行一次 `fsync`，从而分摊开销。

2. 在 SSD（固态硬盘）上，fsync 操作大概有多大开销？

在现代 NVMe SSD 上，单次 `fsync` 的典型延迟大约是 **0.1–2 毫秒**（100–2000 微秒），具体取决于设备型号、固件以及待刷写的数据量。这意味着单线程下你每秒最多大约能做 **500–10,000 次 fsync**。在较老的 SATA SSD 上，`fsync` 延迟可能更高，达到 2–10 毫秒。虽然绝对值看起来不大，但如果每次写一个 key 就 fsync 一次，你会把写入速度限制在约 5,000 ops/s，而有 buffer 的情况下轻松达到百万级。

3. 什么时候可以告诉用户他们的修改（put/delete）已经被持久化了？

只有在 `sync()` 成功返回之后才能保证数据已持久化。具体来说，需要完成两步：
1. 调用 `BufWriter` 的 `flush()`——将数据从用户态的缓冲区刷到内核的 page cache
2. 调用底层文件的 `sync_all()`——触发 `fsync`，确保数据真正落盘

在 `sync()` 被调用之前，数据可能只存在于 `BufWriter` 的内存缓冲区中，或者只在内核的 page cache 中——这两种情况下，一旦断电/崩溃，数据都会丢失。所以结论是：**`put`/`delete` 只有在后续 `sync()` 成功返回后才算是持久化的。**

4. 如何处理 WAL 中的损坏数据？

- 使用 CRC 进行校验
- 从损坏处进行截断，如果 recover 时候出现了损坏的数据块就停止进行恢复。损坏的地方一般都是发生在崩溃时，崩溃时候写入就不完整
- 使用 `key_len` 和 `value_len` 记录字段长度

5. 是否可以设计一个不用 WAL 的 LSM 引擎（比如把 L0 当 WAL 用）？这样设计会有什么影响？

可以，这种设计有时被称为"无 WAL 模式"或"flush-on-write"。核心思路是：每次写入都立即（或非常频繁地）将 memtable 刷写成 L0 SST，让 L0 SST 本身承担持久化的职责。

各方面的影响：

| 方面 | 影响 |
|------|------|
| 写放大 | 显著增大——每个小批次都变成一个 SST，产生大量碎片化的 L0 文件需要 compaction |
| 读性能 | 变差——compaction 来不及消化之前，L0 中会有大量重叠的 SST 需要搜索 |
| 空间放大 | 可能更高——大量小型重叠的 L0 文件 |
| 恢复逻辑 | 更简单——不需要 WAL 恢复逻辑，只需读 manifest 和已有的 SST |
| 写延迟 | 单次写延迟更高——flush 成 SST 比追加写 WAL 开销大得多 |
| Compaction 压力 | 需要更激进的 compaction 来控制 L0 的文件数量 |

本质上，这种设计是用写性能和写放大来换取更简单的恢复逻辑。一些现实中的系统（如 PebbleDB 在某些配置下）也提供了"无 WAL"模式，适用于数据可以从其他来源重建、或者应用层自己做了持久化保障的场景。
